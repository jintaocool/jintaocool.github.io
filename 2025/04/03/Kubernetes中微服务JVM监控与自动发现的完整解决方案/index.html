以下是针对 Kubernetes 中微服务 JVM 监控与自动发现的完整解决方案，结合 Prometheus 的动态发现机制和 Spring Boot 的监控能力，解决 Pod IP 动态变化和大规模部署的挑战：

一、服务端配置（Spring Boot 微服务）

1. 依赖配置（pom.xml）

作用：通过 Actuator 暴露 /actuator/prometheus 端点，提供 JVM 内存、GC、线程池等指标。

2. 应用配置（application.yml）

关键点：

  • 独立端口（如 18081）避免业务端口暴露 Actuator 接口。
  • 通过 ${HOSTNAME} 注入 Pod 名称，便于在 Grafana 中区分实例。

三、Prometheus 自动发现配置

1. Prometheus 配置文件（prometheus.yml）

核心逻辑：

  • 通过 kubernetes_sd_configs 动态发现 Pod。
  • 使用 relabel_configs 重构抓取地址和路径，适配动态 IP。

四、Grafana 监控大屏

1. 数据源配置

  • 添加 Prometheus 数据源，指向 Prometheus 服务地址。

2. 仪表盘导入

  • 使用社区模板（如 ID 13657 或 9568），直接导入 JVM 监控模板。
  • 关键指标：
    • jvm_memory_used_bytes：堆内存使用量
    • jvm_gc_pause_seconds：GC 停顿时间
    • http_server_requests_seconds：接口响应时间分布

五、安全与优化建议

  1. 访问控制：

    • 限制 Actuator 端口的网络策略，仅允许 Prometheus 和内部管理访问。
    • 启用 Spring Security 对 /actuator 路径进行鉴权。

  2. 性能优化：

    • 调整 scrape_interval 避免高频抓取影响服务性能。
    • 使用 Prometheus 的 metric_relabel_configs 过滤非关键指标。

总结

通过上述配置，可实现：

  1. 动态发现：自动适配 Pod IP 变化，支持跨命名空间的大规模部署。
  2. 完整监控：覆盖 JVM 内存、线程、GC 及业务接口性能。
  3. 安全隔离：独立监控端口和权限控制，降低风险。
